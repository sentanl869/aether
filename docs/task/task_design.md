# Aether 设计文档任务跟踪（task_design）

## 1. 目标

- 用于跟踪 `docs/design.md` 各章节编写进度与完成标准。
- 以 `docs/requirements.md` 作为唯一需求基线，确保需求可追踪、可验收、可指导编码实现。

## 2. 全局约束（强制）

- C-001（画图约束）：所有需要画图的任务，必须使用 `mermaid` 语法；禁止使用图片、PlantUML 或其他图形语法。
- C-002（可编码约束）：所有任务完成后，`docs/design.md` 必须达到“可直接指导编码”的粒度，至少覆盖：数据结构、状态流转、接口契约、异常码、并发/幂等、关键流程时序、边界条件。
- C-003（API 约束）：所有 API 接口设计必须遵循 OpenAPI 3.0 规范与 RESTful 风格（资源化路径、标准 HTTP Method、统一响应结构、错误码一致性、幂等与并发控制）。

## 3. 任务拆分与依赖

| Task ID | 任务名称 | 章节范围（design.md） | 依赖 | 状态 |
| --- | --- | --- | --- | --- |
| T1 | 基线与追踪框架 | 2、14.4、15.2 | 无 | TODO |
| T2 | 架构与核心模型 | 3、5、6 | T1 | TODO |
| T3 | 契约与执行编排 | 7、8 | T2 | TODO |
| T4 | 功能域与 K8s 编排 | 4、9 | T2、T3 | TODO |
| T5 | 安全、隔离、可运维与 NFR | 10、11、12、13 | T3、T4 | TODO |
| T6 | 验收与交付收口 | 14（除 14.4）、15（除 15.2） | T1~T5 | TODO |

说明：

- 推荐按 `T1 -> T2 -> T3 -> T4 -> T5 -> T6` 顺序推进。
- 若并行执行，必须先冻结跨任务共享基线：状态枚举、错误码、资源命名、审计字段、任务模型。

## 4. 任务明细（DoD 驱动）

### T1 基线与追踪框架

- 目标：建立需求到设计的追踪骨架，先消除“写完后无法审计覆盖率”的风险。
- 对应章节：`2`、`14.4`、`15.2`
- 输出物：
- `2` 章完成范围、边界、原则与约束落盘。
- `14.4` 提供逐条需求 ID 覆盖核对表（R-ENV~R-OPS、NFR）。
- `15.2` 定义需求变更进入设计的流程与版本规则。
- DoD：
- 每个需求簇至少映射到一个设计章节。
- 明确“已覆盖/待补充/不适用”三类状态。
- 与 `docs/requirements.md` 无冲突陈述。
- 进度：
- [ ] 完成

### T2 架构与核心模型

- 目标：定义统一抽象、领域模型、状态机，作为后续 API 与执行链路的基础。
- 对应章节：`3`、`5`、`6`
- 输出物：
- 系统上下文图、逻辑架构图、核心时序图（全部 `mermaid`）。
- ERD 与关键实体字段、唯一性约束、软删与审计策略。
- 一级资源、支撑资源、任务、发布制品的状态机定义。
- DoD：
- `R-ABS`、`R-DATA`、状态机相关设计输入全部有落点。
- 状态迁移触发条件、失败分支、补偿入口可追踪。
- 模型字段可直接映射到数据库表与索引策略。
- 进度：
- [ ] 完成

### T3 契约与执行编排

- 目标：固化 API 契约与异步任务机制，形成统一执行面。
- 对应章节：`7`、`8`
- 输出物：
- OpenAPI 3.0 风格接口分组、路径规范、请求/响应 schema、错误模型。
- `Idempotency-Key`、`resource_version`、串行化键、重试/超时/取消/补偿机制。
- 关键流程时序图（`mermaid`）。
- DoD：
- 所有 CUD 接口满足异步任务提交，Query 保持同步语义。
- 错误码最小集合与业务错误场景一一对应。
- 能支持代码生成或直接指导 handler/service/repository 分层实现。
- 进度：
- [ ] 完成

### T4 功能域与 K8s 编排

- 目标：把 10 个功能域需求落到可执行设计，明确模板、制品、部署与回滚细节。
- 对应章节：`4`、`9`
- 输出物：
- `4.1~4.10` 各资源域的创建/更新/删除/查询/发布规则与边界。
- Helm 模板路线、values 合并、镜像与凭证校验、命名标签规范。
- 部署/升级/回滚/漂移处理流程图（`mermaid`）。
- DoD：
- `R-ENV~R-OPS` 需求逐条可映射到章节条目。
- 共享资源与嵌入式资源在模型、API、权限、展示上隔离清晰。
- 可以直接指导编写各资源适配器与编排执行器。
- 进度：
- [ ] 完成

### T5 安全、隔离、可运维与 NFR

- 目标：补全非功能与生产可用性约束，确保上线可治理、可审计、可扩展。
- 对应章节：`10`、`11`、`12`、`13`
- 输出物：
- 权限模型、租户隔离、Secret 生命周期、审计策略。
- 日志/指标/追踪/告警方案与运维手册。
- 性能预算、容量规划、压测与验收方案。
- DoD：
- 覆盖 `S-SEC` 与 `NFR-001~019` 的设计响应。
- 指标口径、采样维度、告警阈值具备可实施性。
- 关键高风险动作均可审计追溯。
- 进度：
- [ ] 完成

### T6 验收与交付收口

- 目标：将设计文档收口成“可验收、可交付、可研发分解”的最终版本。
- 对应章节：`14`（不含 `14.4`）、`15`（不含 `15.2`）
- 输出物：
- 需求 ID -> 测试场景 -> 验收标准矩阵。
- 发布与回滚计划、交付清单、开放问题清单。
- 设计自检记录（冲突、遗漏、待决策项）。
- DoD：
- 形成端到端验收路径：需求 -> 设计 -> 用例 -> 发布。
- `docs/design.md` 完整性达到“指导编码”标准（满足 C-002）。
- 未决问题有明确 owner 与关闭条件。
- 进度：
- [ ] 完成

## 5. 统一完成判定（Definition of Done）

- [ ] `docs/design.md` 1~15 章全部完成，且与 `docs/requirements.md` 一致。
- [ ] 所有图均为 `mermaid`，渲染可读、语义完整。
- [ ] API 设计符合 OpenAPI 3.0 + RESTful，并落地统一响应与错误码规范。
- [ ] 需求覆盖审计完成，关键需求簇无“未覆盖”项。
- [ ] 文档达到可指导编码级别：研发可据此拆分模块并开始实现。
