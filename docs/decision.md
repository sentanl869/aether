# Aether 需求阶段决策记录

## D-001 Aether 需求基线聚焦部署能力

- 日期：2026-02-28
- 状态：已采纳
- 决策：`requirements.md` 不再描述“生成高代码应用 Helm Chart 并发布版本到镜像仓库”流程，Aether 需求仅保留部署阶段对镜像或 Chart 引用的消费与管理能力。
- 原因：该流程属于上游制品工程链路，在 Aether 需求基线中保留相关表述会引入跨模块职责耦合与实现假设。
- 影响：
  - `requirements.md` 中文档目标、非目标、边界集成、验收标准对应条目已移除该流程描述。
  - Aether 接口继续接收“产物引用”而非“产物构建参数”，核心 CUD/Query 关注部署编排本身。

## D-002 统一接口模型：CUD 异步、Query 同步

- 日期：2026-02-27
- 状态：已废止（被 D-011 取代）
- 决策：Create/Update/Delete 全部返回 `ACCEPTED + task_id`，实际部署由异步任务执行；Query 同步返回并支持分页/过滤/排序。
- 原因：与需求中“CUD 统一异步、Query 同步”完全对齐，且适配 Helm/K8S 长耗时操作。
- 影响：
  - 上层模块必须具备任务状态轮询/订阅能力。
  - Aether 必须提供任务查询接口与统一状态机。

## D-003 采用“类型注册 + 通用 Helm 执行器”扩展机制

- 日期：2026-02-27
- 状态：已采纳
- 决策：通过资源类型注册机制承载模板映射与参数结构校验规则；部署、更新、删除流程由通用执行器复用。
- 原因：需求要求无缝扩展新资源类型并复用部署行为，避免按资源类型复制业务流水线。
- 影响：
  - 新增资源类型主要通过新增类型定义与规则完成。
  - 核心部署编排逻辑保持稳定，降低维护成本。

## D-004 数据服务组件可见性分层

- 日期：2026-02-27
- 状态：已废止（被 D-015 取代）
- 决策：数据服务组件区分 `USER_MANAGED` 与 `EMBEDDED_PRIVATE` 两类可见性。
- 原因：需求明确低代码平台/记忆体/Helm 应用自带依赖数据服务不能进入通用数据服务组件统一管理视图。
- 影响：
  - 查询接口必须按可见性规则过滤结果。
  - 删除父资源时，其 `EMBEDDED_PRIVATE` 依赖强制级联删除，不提供保留策略。

## D-005 权限边界按工作空间统一

- 日期：2026-02-27
- 状态：已采纳
- 决策：同一工作空间内不做按创建者隔离；Aether 以工作空间为核心授权边界。
- 原因：需求明确“同一工作空间内没有更细粒度权限隔离”。
- 影响：
  - 资源查询与操作校验以 `workspace_id` 为第一维度。
  - 角色（超级管理员/空间管理员/普通用户）差异主要体现在可操作资源集合上。

## D-006 异步任务基础设施固定为 PostgreSQL + Redis

- 日期：2026-02-27
- 状态：已废止（被 D-011 取代）
- 决策：任务与资源元数据落 PostgreSQL；任务队列与分布式锁使用 Redis。
- 原因：技术栈为强约束要求。
- 影响：
  - 任务执行需内置幂等键与锁键规范。
  - 需设计重试和故障恢复策略，满足任务可靠性目标。

## D-007 创建请求部署位置必填

- 日期：2026-02-27
- 状态：已采纳
- 决策：资源创建请求必须显式携带部署位置，部署位置为目标 K8S 集群（`cluster_id`）且为必填参数。
- 原因：澄清结果明确“部署位置必选”，避免多集群关联下的目标不确定性。
- 影响：
  - Aether 统一 CUD 入参需将 `cluster_id` 设为强校验必填字段。
  - 未传递部署位置的请求必须直接拒绝。

## D-008 低代码平台“单实例”约束范围

- 日期：2026-02-27
- 状态：已废止（被 D-010 取代）
- 决策：同一工作空间内“每种内置低代码平台最多一个实例”的约束，仅适用于内置实例，不适用于用户导入的同类型外部 Chart 实例。
- 原因：澄清结果明确该约束范围不包含外部导入 Chart。
- 影响：
  - 该约束不再作为 Aether 需求基线中的校验职责。
  - 若业务仍需该规则，应由上层模块在调用 Aether 前执行判断。

## D-009 Query 排序规则责任边界

- 日期：2026-02-27
- 状态：已废止（被 D-012 取代）
- 决策：Query 排序字段范围与默认排序规则由调用方传入参数决定，Aether 只提供排序能力支持，不内置固定白名单和默认规则。
- 原因：澄清结果明确排序策略应由调用方控制。
- 影响：
  - Aether 查询接口需支持动态排序参数透传与执行。
  - 排序策略演进主要在调用方，不强耦合 Aether 内部实现。

## D-010 资源级业务约束校验责任上移

- 日期：2026-02-28
- 状态：已采纳
- 决策：Aether 需求基线移除资源级业务约束校验职责（如单实例限制、资源来源组合限制）；该类校验由上层业务模块在下发资源操作前完成。
- 原因：资源级业务约束属于业务域规则，放在 Aether 会引入跨模块耦合并降低通用部署引擎复用性。
- 影响：
  - `requirements.md` 中资源章节调整为“部署形态说明”，不再要求 Aether 拒绝此类业务约束冲突。
  - 上层模块需在调用 Aether 前完成前置校验，再将通过校验的统一请求下发。

## D-011 Aether 采用“无状态部署执行 + 上层维护状态”边界

- 日期：2026-02-28
- 状态：已采纳
- 决策：Aether 仅提供基础部署执行与运行态查询能力，不再承担资源/任务状态持久化；业务方通过轮询 Aether Query 接口自行维护状态表与历史快照。
- 原因：当前阶段优先收敛 Aether 责任边界，降低部署引擎复杂度与存储耦合，避免在基础设施层重复建设业务状态模型。
- 影响：
  - `requirements.md` 移除“任务元数据与资源元数据必须落 PostgreSQL”“必须提供任务查询接口”等强约束，改为“Query 返回当前观测态”。
  - CUD 返回由 `task_id` 模型调整为 `request_id` 回显 + `ACCEPTED` 受理语义；状态流转由上层轮询资源查询接口自行维护。
  - D-002、D-006 中关于“任务持久化与任务查询”的约束失效，以本决策为准。

## D-012 Query 收敛为 `task_id` 单任务查询

- 日期：2026-02-28
- 状态：已采纳
- 决策：Aether Query 能力从“分页/过滤/排序的通用列表查询”收敛为“仅按 `task_id` 的单任务查询”；CUD 受理返回需包含 `task_id` 供上层轮询。
- 原因：当前阶段任务查询目标聚焦为“查询特定部署任务当前状态与结果”，不再承载任务列表检索能力，降低接口复杂度并收敛模块边界。
- 影响：
  - `requirements.md` 的 FR-API-004、FR-Q-002、AC-003 等条目已调整为 `task_id` 单查模型，并移除分页/过滤/排序要求。
  - CUD 受理返回字段按本次决策明确为 `request_id + task_id + ACCEPTED`，覆盖 D-011 中仅 `request_id` 回显的旧表述。
  - D-009（Query 排序规则责任边界）失效，查询排序相关约束不再适用。
  - 上层模块需自行维护并持有 `task_id`，通过单任务轮询完成业务状态推进与历史快照沉淀。

## D-013 幂等键仅允许使用已定义字段

- 日期：2026-02-28
- 状态：已采纳
- 决策：幂等约束从 `request_id + workspace_id + action + resource_key` 收敛为 `request_id + workspace_id + action`；移除未在接口契约中定义来源的 `resource_key`。
- 原因：`requirements.md` 未定义 `resource_key` 的来源与映射规则，继续保留会导致实现歧义与跨模块解释不一致。
- 影响：
  - `requirements.md` 的 FR-EXEC-003 已改为仅依赖已定义入参字段构造幂等键。
  - `requirements.md` 的 NFR-003 重试描述同步改为复用同一 `request_id + workspace_id + action`。

## D-014 任务拆解补充基线并入需求文档

- 日期：2026-03-01
- 状态：已废止（被 D-017 取代）
- 决策：在 `requirements.md` 新增“设计与任务拆解补充基线（规范性附录）”，固定以下内容为规范输入：生效决策一页基线、接口契约附录、任务状态机与幂等规则、资源类型注册 schema 与接入模板、依赖表达及删除策略参数、验收用例矩阵。
- 原因：原需求条目边界清晰但实现粒度不足，无法直接拆解为可执行设计任务与开发任务。
- 影响：
  - 设计与任务拆解应以 `requirements.md` 第 13 章为最小实现输入，不再依赖口头补充。
  - 需求 -> 设计 -> 任务 -> 测试可基于统一条目直接建立追踪关系。
  - 接口、状态、错误码与测试覆盖范围在需求阶段前置收敛，降低后续反复澄清成本。

## D-015 依赖边界与删除策略以显式请求为准

- 日期：2026-03-01
- 状态：已采纳
- 决策：废止 D-004 的“内建依赖可见性分类 + 强制级联删除”约束；Aether 对依赖仅处理请求内显式声明项，删除行为由 `delete_options.dependency_policy` 与 `dependency_ids` 显式驱动。
- 原因：D-004 与当前需求基线（FR-DM-003、FR-DM-004、AC-004）不一致，会导致 Aether 承担超出边界的依赖展示语义与隐式级联策略。
- 影响：
  - Aether 不维护依赖展示语义模型，不内建基于展示分类的强制级联规则。
  - 依赖删除/保留策略责任边界与参数定义在需求文档中统一落地，供上层模块显式传参。
  - 生效决策基线中，D-004 归入废止清单，不再作为设计输入。

## D-016 幂等与重入采用“无存储 + 可观测事实判定”

- 日期：2026-03-01
- 状态：已采纳
- 决策：Aether 在幂等与重入场景下不引入资源表/任务表/幂等记录表；判定逻辑基于“幂等键与请求指纹的确定性计算”以及 Helm/K8S 当前可观测事实（release 元数据、revision、资源注解）。
- 原因：需求边界已明确 Aether 无状态，不承担持久化；幂等/重入需要在无存储约束下给出可执行判定规则。
- 影响：
  - `requirements.md` 第 13.3.2 已改为“无状态前提 + 事实来源 + 命中规则 + 重入规则”。
  - 幂等重试由调用方复用同一 `request_id + workspace_id + action` 发起，Aether 基于当前观测态决定“直接返回”或“继续执行”。
  - 该规则与 D-011、D-012、D-013 一致，作为后续设计与任务拆解输入。

## D-017 接口定义与 OpenAPI 定义归属设计文档

- 日期：2026-03-01
- 状态：已采纳
- 决策：`requirements.md` 不再承载函数接口定义、DTO/枚举字段定义、OpenAPI 定义等设计产物；需求文档仅保留能力边界与约束。上述定义必须在 `docs/design.md` 中完整提供，并由 `docs/task/task_design.md` 建立任务追踪。
- 原因：函数接口与 OpenAPI 属于“如何设计”的范畴，放在需求文档会造成阶段边界混淆，降低需求基线稳定性。
- 影响：
  - D-014 中“将接口契约附录写入需求文档”的做法失效，以本决策为准。
  - `requirements.md` 第 13.2 调整为“设计文档强制约束”，明确设计阶段必须交付的接口定义与映射内容。
  - 需求评审与设计评审的职责边界更清晰：需求评审关注能力与边界，设计评审关注接口与实现细节。

## D-018 设计文档图示语法与完成粒度约束

- 日期：2026-03-01
- 状态：已采纳
- 决策：设计阶段所有需要画图的任务必须使用 Mermaid 语法，禁止使用图片、PlantUML 或其他图形语法；所有任务完成后 `docs/design.md` 必须达到“可直接指导编码”的粒度。
- 原因：统一图示语法可降低评审与维护成本；明确完成粒度可避免设计文档停留在概念层，导致实现阶段反复补充。
- 影响：
  - `requirements.md` 第 13.2 已新增图示语法约束与设计完成粒度约束。
  - 设计评审需将 Mermaid 合规性与“可直接编码”覆盖项作为必检项。
  - `docs/task/task_design.md` 任务验收需检查 `docs/design.md` 对数据结构、状态流转、接口契约、异常码、并发/幂等、关键流程时序、边界条件的覆盖完整性。

## D-019 Aether 增加 values.yaml 渲染能力并要求 Helm 执行语义等效

- 日期：2026-03-01
- 状态：已采纳
- 决策：Aether 必须具备将调用方参数渲染为 `values.yaml` 的能力，输出形态至少支持“YAML 数据”与“可执行输入文件”；Create/Update 执行语义必须等效于 `helm install/upgrade -f values.yaml`。
- 原因：Aether 实际部署行为需要与 Helm 标准命令保持一致，避免“接口参数解释路径”和“Helm 执行输入路径”分叉导致行为偏差。
- 影响：
  - `requirements.md` 新增 FR-CORE-004、FR-API-006、FR-EXEC-004、NFR-005、AC-005、AC-006。
  - `requirements.md` 的错误模型补充“渲染错误”类别，用于覆盖 values 生成失败或不满足约束的场景。
  - 设计文档必须新增 values 渲染规则、输出形态、渲染一致性策略与 Helm 等效性保证机制。
