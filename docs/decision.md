# Aether 部署模块决策记录

> 记录需求分析过程中的关键决策与理由，便于后续实现和复盘。

## 2026-01-28

### ADR-001: 组件实例与 Agent 应用解耦

- 决策：组件实例与 Agent 应用分开管理，应用仅关联已有组件实例。
- 原因：提升复用能力，减少重复部署；便于统一运维与升级。
- 影响：应用创建流程需支持组件选择与关系保存。

### ADR-002: 组件服务名（Service Name）默认由平台生成

- 决策：组件实例的 svc 名称默认由平台生成，用户不手填。
- 原因：避免命名冲突，提升一致性与自动化程度。
- 影响：平台需在组件实例详情中展示生成的服务名与访问地址。
- 状态：已由 ADR-009 细化替代。

### ADR-003: 连接信息注入优先使用环境变量 + Secret

- 决策：Agent 与组件的连接信息默认通过环境变量注入；敏感信息使用 Secret。
- 原因：K8S 原生支持，易于实现与运维；降低配置复杂度。
- 影响：部署模板需支持 ConfigMap/Secret 生成与挂载。

### ADR-004: 应用删除默认不级联删除组件实例

- 决策：删除应用时不自动删除组件实例（默认保留）。
- 原因：组件可能被多个应用复用，级联删除风险高。
- 影响：应用删除需明确提示组件实例保留策略；可提供可选级联删除。
- 状态：已由 ADR-012 细化替代。

### ADR-005: 模板系统优先支持 Helm 或参数化 YAML

- 决策：初期采用 Helm 或参数化 YAML 模板（择一落地）。
- 原因：复用成熟生态；参数化能力强。
- 影响：实现时需评估当前团队熟悉度与平台已有实现方式。

### ADR-006: 镜像仓库凭证由平台统一管理并下发

- 决策：Aether 需管理镜像仓库凭证并下发至目标命名空间（用于推送/拉取）。
- 原因：镜像上传与部署均依赖凭证，否则无法完成推送/拉取。
- 影响：需落地 Secret 管理与关联到 ServiceAccount 或 Pod ImagePullSecrets。
- 已确认：平台级统一凭证。

### ADR-007: 组件运行时深度运维纳入范围

- 决策：组件实例的扩缩容、升级/回滚、备份/恢复等能力纳入需求范围。
- 原因：平台需要对组件提供完整生命周期管理能力。
- 影响：模板与控制面需预留运维入口与流程，可能引入额外作业（Job/CronJob）。

### ADR-008: 用户隔离为硬性需求，具体网络隔离方案待定

- 决策：用户间组件默认不可互通通信，需提供隔离能力。
- 原因：多租户安全要求，避免跨用户访问。
- 影响：采用 Namespace + RBAC；是否追加 NetworkPolicy 需进一步评估。

### ADR-009: 组件 Service 命名为条件生成

- 决策：若 Agent 部署可自动关联组件实例，Service 名称由平台生成；否则允许用户自定义。
- 原因：在自动关联时可避免冲突；在无法关联时需满足外部依赖。
- 影响：组件创建表单需支持条件字段与校验。

### ADR-010: 连接信息注入细化为环境变量 + Secret

- 决策：Agent 连接组件时，用户名、密码、地址等写入环境变量；敏感字段通过 Secret 注入。
- 原因：符合 K8S 原生实践，便于权限与审计管理。
- 影响：需评估 Secret 与用户隔离边界的关系。

### ADR-011: 不允许跨 Namespace 关联组件实例

- 决策：组件实例与 Agent 必须位于同一 namespace。
- 原因：简化隔离与权限模型，降低跨租户访问风险。
- 影响：应用部署校验需阻止跨 namespace 关联。

### ADR-012: 应用删除支持可选级联删除

- 决策：应用删除提供“是否级联删除组件实例”的选项。
- 原因：兼顾复用与资源回收需求。
- 影响：删除流程需区分关联关系与删除策略。

### ADR-013: 镜像仓库凭证支持多仓库与多命名空间

- 决策：在平台级统一凭证前提下，支持多仓库、多命名空间下发。
- 原因：满足多环境与多集群的镜像拉取需求。
- 影响：凭证管理需要维护仓库与命名空间映射关系。

### ADR-014: 权限模型待定，需求需显式标注

- 决策：所有涉及权限的能力在需求中标注为待定点，后续统一补齐。
- 原因：权限模型尚未确定，避免假设导致重构。
- 影响：实现阶段需预留权限策略配置与扩展点。

### ADR-015: 组件备份/恢复需覆盖所有组件

- 决策：深度运维中的备份/恢复能力需要覆盖所有内置组件。
- 原因：用户对组件生命周期管理的完整性要求。
- 影响：可能需要为不同组件定制 Job/CronJob 方案或适配器。

## 2026-02-11

### ADR-016: 备份/恢复能力移出当前需求范围

- 决策：在当前阶段移除组件备份/恢复相关全部需求，不纳入实现范围。
- 原因：当前交付目标聚焦部署、关联与运行态运维能力；备份恢复实现复杂度高、组件差异大。
- 影响：`requirements.md` 中所有备份/恢复条目已删除；深度运维仅保留扩缩容、升级/回滚、滚动重启与健康检查。
- 替代关系：覆盖 ADR-007（部分）与 ADR-015。

### ADR-017: 工作空间与 namespace 映射规则固定

- 决策：一个工作空间对应每个关联 K8S 集群中的一个同名 namespace；若工作空间关联多个集群，则在每个集群创建同名 namespace。
- 原因：统一多集群下的租户边界与资源组织方式，降低运维与权限实现复杂度。
- 影响：创建组件实例、Agent 实例、Agent 应用时，必须先选择工作空间与目标集群；namespace 由映射规则确定，不允许自由指定。

### ADR-018: 工作空间内禁止跨集群部署与关联

- 决策：同一工作空间内，应用不可跨 K8S 集群部署，组件不可跨集群关联。
- 原因：保持网络边界、权限边界和故障域一致，避免跨集群依赖导致治理复杂度上升。
- 影响：应用创建/更新流程需增加集群一致性校验；关系持久化中需记录并校验集群维度。

### ADR-019: 权限模型定稿为“超级管理员 + 用户”

- 决策：用户模型分为超级管理员和用户。超级管理员可操作全部资源；用户仅可操作其关联工作空间内资源。
- 原因：满足平台运维与租户隔离的双重需求，并控制权限模型复杂度。
- 影响：移除“权限模型待定”状态；鉴权需按“全局角色 + 工作空间关联”执行。
- 补充：同一工作空间内用户可对该工作空间内所有资源进行 CRUD。
- 替代关系：覆盖 ADR-014。

### ADR-020: 模板技术路线优先级固定

- 决策：组件模板技术路线优先级为 Helm > Kustomize > 原生 YAML。
- 原因：优先采用生态成熟、参数化与版本管理能力更强的方案，兼容不同组件落地形态。
- 影响：模板管理与渲染链路按该优先级实现；需求文档不再将此项列为待澄清问题。
- 替代关系：细化并替代 ADR-005。

### ADR-021: Agent 应用创建支持“一次选择并关联已有组件实例”

- 决策：用户使用自有 Agent 镜像创建应用时，可直接选择已提前创建好的组件实例，并在一次提交中完成应用创建与关联部署。
- 原因：减少多步骤操作，提高应用交付效率，强化组件复用体验。
- 影响：应用创建接口需支持组件实例列表输入与原子性校验。

### ADR-022: 存储配置收敛为仅暴露 StorageClass（SC）

- 决策：在工作空间内创建组件实例、Agent 实例、Agent 应用时，存储配置仅允许选择 StorageClass（SC），不继续向下暴露存储细节。
- 原因：降低用户认知负担，避免暴露底层存储实现复杂度，统一平台交互。
- 影响：创建表单与 API 字段需收敛；底层存储细节由模板默认值或平台策略管理。

### ADR-023: 隔离边界以工作空间为单位

- 决策：将隔离粒度确定为工作空间。不同工作空间资源默认不可互通；同一工作空间内资源可按业务需要互通。
- 原因：与“工作空间内用户可对所有资源 CRUD”的授权模型保持一致，避免用户级隔离与空间级授权冲突。
- 影响：`requirements.md` 中“用户间不可互通”调整为“工作空间间不可互通”。
- 替代关系：细化并替代 ADR-008 的隔离粒度描述。

### ADR-024: 当前阶段不强制 NetworkPolicy

- 决策：隔离实现当前采用 Namespace + RBAC，不强制启用 NetworkPolicy。
- 原因：先保障权限与资源边界落地，降低首期实施复杂度。
- 影响：需求文档移除对此项的待澄清状态；后续如引入 NetworkPolicy 作为能力增强另行立项。

### ADR-025: Secret 边界与可见性规则定稿

- 决策：Secret 必须限定在工作空间对应 namespace 内；普通用户不可查看明文，仅可引用；仅平台控制面服务账号可读取明文用于注入。
- 原因：兼顾安全合规与工作空间隔离要求。
- 影响：创建/更新流程、鉴权策略和审计日志需体现“可引用不可明文读取”。
- 替代关系：细化 ADR-010 中 Secret 边界未定部分。

### ADR-026: 工作空间与集群解绑采用自动回收策略

- 决策：工作空间与集群解绑时，自动删除该工作空间在目标集群对应 namespace 及其全部资源。
- 原因：避免解绑后残留资源导致运维负担与安全风险。
- 影响：该操作仅允许超级管理员执行，且需二次确认。

### ADR-027: Agent 实例作为独立资源提供 CRUD

- 决策：Agent 实例从“应用内部对象”调整为独立资源，提供创建/更新/删除/查询能力。
- 原因：满足复用、预创建与生命周期独立管理需求。
- 影响：接口层新增 Agent 实例 CRUD；应用创建支持“选择已有 Agent 实例”与“一次性创建并关联”两种模式。

### ADR-028: 应用删除策略确认

- 决策：应用删除默认不级联删除组件实例；支持显式选择级联删除。
- 原因：平衡组件复用与资源回收需求。
- 影响：删除流程需明确展示删除影响范围并进行参数化控制。
- 替代关系：确认并沿用 ADR-012，作为最终策略。

### ADR-029: 存储配置调整为 SC + 容量

- 决策：在工作空间内创建组件实例、Agent 实例、Agent 应用时，存储配置允许选择 StorageClass（SC）和容量（Gi）。
- 原因：在保持抽象简化的同时，满足最基本容量差异化需求。
- 影响：表单/API 字段从“仅 SC”调整为“SC + 容量”，仍不暴露底层存储细节。
- 替代关系：替代 ADR-022。

### ADR-030: 组件 Service 名称策略保持条件式

- 决策：组件 Service 命名采用条件策略：
  - 在“应用创建时直接关联组件实例”场景下由平台自动生成。
  - 在“独立创建组件实例”且用户显式选择自定义时允许手填（同 namespace 唯一）。
- 原因：兼顾自动化命名一致性与独立组件场景下的命名可控性。
- 影响：组件创建与应用创建流程均需明确命名来源与唯一性校验。
- 替代关系：细化并延续 ADR-009。

### ADR-031: 部署模块交付接口移除“工作空间与集群绑定管理”

- 决策：`requirements.md` 的“交付物与接口（建议）”中移除“工作空间与集群绑定管理接口（含 namespace 同步状态）”。
- 原因：该能力属于平台内部管理能力，不纳入部署模块对外交付接口范围。
- 影响：部署模块交付接口聚焦 Agent 实例、组件模板/实例、Agent 应用及应用关系查询；工作空间-集群绑定与 namespace 同步由内部模块承接。

## 2026-02-19

### ADR-032: 部署模块后端技术栈定稿为 Go + Gin + PostgreSQL

- 决策：Aether 部署模块后端必须使用 Go 开发，Web/API 框架固定为 Gin，持久化数据库固定为 PostgreSQL（Postgres）。
- 原因：统一团队实现技术栈，降低工程协作与运维复杂度，避免多技术路线并行导致交付风险。
- 影响：`requirements.md` 新增“实现约束（技术栈）”章节，作为强制实现边界。

### ADR-033: Agent 实例、组件实例、Agent 应用的 CRUD 统一采用异步生产者-消费者模型

- 决策：上述三类资源的创建、更新、删除、查询全部通过统一任务队列执行；控制面负责生产任务，任务执行器负责消费任务并异步落地。
- 原因：统一执行模型，解耦 API 请求与实际部署/变更动作，提升系统吞吐与可恢复性。
- 影响：提交 CRUD 请求后统一返回 `task_id`；新增任务状态/结果查询接口；任务执行需满足幂等、重试和同资源串行化控制。

### ADR-034: 查询接口改为同步，异步仅用于变更类操作

- 决策：Agent 实例、组件实例、Agent 应用的查询接口改为同步返回；异步任务模型仅覆盖创建、更新、删除（CUD）操作。
- 原因：查询属于读操作，改为同步可降低调用与前端交互复杂度，提升可观测与排障效率。
- 影响：保留任务队列用于 CUD；查询接口直接读取数据库与集群状态聚合结果。
- 替代关系：覆盖 ADR-033 中“查询异步”的部分。

### ADR-035: 工作空间网络隔离方案评估与取舍

- 决策：放弃将“不同工作空间实例默认不可互通”作为当前阶段硬性网络隔离要求，并移除相关网络隔离表述。
- 原因：
  - 多集群场景下 NetworkPolicy 需在每个集群独立生效，依赖各集群 CNI 的一致支持与正确执行。
  - 平台通过 kubeconfig 纳管异构集群，无法在当前范围内保证所有集群具备可验证的一致网络策略执行能力。
  - 在该前提下将其写为硬性要求，存在“需求可写但不可稳定验收”的落地风险。
- 影响：当前隔离边界以工作空间级权限与资源边界控制为主（namespace 映射、RBAC、跨边界关联限制）；网络强隔离后续单独立项。
- 替代关系：覆盖 ADR-023 中“默认不可互通通信”的网络表述，并替代 ADR-024。

### ADR-036: 工作空间与集群解绑改为“冻结 + 校验 + 可恢复窗口”

- 决策：解绑流程不再直接删除 namespace；改为“先冻结、再校验、进入可恢复窗口、窗口结束后执行回收”。
- 原因：降低误操作导致的不可逆数据丢失风险，保证生产场景下的可恢复性。
- 影响：
  - 冻结后禁止新增部署/变更任务。
  - 校验需覆盖运行中任务与关键资源状态。
  - 可恢复窗口内允许撤销解绑并恢复服务。
  - 窗口结束后再执行 namespace 及关联资源回收。
- 替代关系：替代 ADR-026。

### ADR-037: Application 与 Agent 实例关系定稿为一对一

- 决策：Application 与 Agent 实例为一对一关系；组件实例允许被多个 Application 复用。
- 原因：明确生命周期边界，避免应用与 Agent 复用关系导致删除与运维语义混乱。
- 影响：应用创建/更新需校验 Agent 实例唯一归属；组件实例关系需维护引用计数或等价引用关系。
- 替代关系：细化 ADR-027 的资源关系约束。

### ADR-038: 级联删除范围扩展

- 决策：当用户显式选择级联删除时，删除范围包含关联组件实例、Agent 实例、PVC、Secret、ConfigMap（及其与应用绑定的派生资源）。
- 原因：级联删除语义需覆盖应用运行依赖，避免残留资源。
- 影响：删除流程需提供影响面预览、依赖校验与失败回滚/补偿机制。
- 替代关系：细化 ADR-028。

### ADR-039: Secret 更新支持版本化引用

- 决策：Secret 更新采用版本化引用机制（新版本写入并切换引用），保留历史版本用于回滚与审计。
- 原因：满足敏感配置安全更新、灰度切换与可追溯需求。
- 影响：应用配置中需记录 Secret 引用版本；权限与审计需覆盖版本级操作。
- 替代关系：细化 ADR-025。

### ADR-040: 工作空间内不引入角色分层

- 决策：当前阶段不引入工作空间内 Owner/Maintainer/Viewer 等角色分层。
- 原因：保持权限模型简单，与既有“超级管理员 + 用户”模型一致。
- 影响：同一工作空间内用户继续共享 CRUD 权限；细粒度授权后续如需单独立项。
- 替代关系：延续 ADR-019 的权限收敛方向。

### ADR-041: 验收标准与测试用例由需求方提供，需求持续演进

- 决策：验收标准与测试用例由需求方提供；需求不设置冻结点，允许持续变化。
- 原因：保持需求迭代灵活性，并由业务侧掌握验收口径。
- 影响：实现与评审流程需接受增量变更；每轮变更需同步更新需求与决策记录，确保可追溯。

## 2026-02-21

### ADR-042: 需求文档新增“设计输入待决项”清单并设置关闭时限

- 决策：将“无待澄清问题”调整为“设计输入待决项”，按编号、截止日期、影响范围管理。
- 原因：当前已进入设计准备阶段，需显式暴露关键未决点，避免在详细设计阶段隐性返工。
- 影响：详细设计评审前需先关闭待决项；需求评审增加待决项清零检查。

### ADR-043: 领域模型与唯一性约束在需求层固化

- 决策：在 `requirements.md` 固化核心实体、主标识、唯一约束、软删策略和关系完整性规则。
- 原因：ERD、索引、接口幂等与并发控制均依赖统一数据约束，不能留到实现期临时决定。
- 影响：后续数据库设计必须遵循该约束集；命名规则与跨资源引用在接口层提前校验。

### ADR-044: 共享组件的级联删除冲突返回 409，不执行强制删除

- 决策：删除 Application 且 `cascade=true` 时，若组件仍被其他应用引用，返回 `409 Conflict` 并中止该组件删除。
- 原因：组件复用是核心能力，强制删除会导致其他应用不可预期中断。
- 影响：删除流程需先做引用计数校验；前端需展示冲突详情并引导用户先解除引用。
- 替代关系：细化 ADR-038 的级联删除执行条件。

### ADR-045: 异步任务模型补齐幂等、串行、重试、超时、取消语义

- 决策：CUD 请求强制 `Idempotency-Key`，同键 24h 去重；同资源串行化执行；固定重试退避与超时、取消规则。
- 原因：仅定义“异步”不足以支撑稳定实现，需要把可恢复性与一致性约束前置到需求层。
- 影响：任务队列与执行器需实现去重、锁键、重试和取消能力；API 契约同步增加相关字段与错误处理。
- 替代关系：细化 ADR-033 与 ADR-034 的执行语义。

### ADR-046: 接口契约与并发控制要求前置到需求文档

- 决策：新增统一 API 契约要求（前缀、响应结构、分页过滤、错误码、幂等、并发控制）。
- 原因：接口标准化是前后端并行开发与测试自动化的前提。
- 影响：后续 OpenAPI 设计以该约束为基线；并发控制策略需在待决项关闭后统一落地。

### ADR-047: 非功能需求量化并建立需求-测试追踪矩阵

- 决策：将 NFR 从原则描述升级为量化指标，并新增需求 ID 到验收用例追踪矩阵。
- 原因：缺乏量化指标无法进行性能/可用性验收，缺乏追踪关系会导致测试覆盖失焦。
- 影响：测试与 SRE 需基于量化阈值执行压测和验收；需求变更需同步更新追踪矩阵。
- 替代关系：替代 ADR-041 中“验收由需求方单方提供”的执行口径。

### ADR-048: Agent 制品扩展为 Image 与 Helm Chart 双类型

- 决策：Agent 创建来源不再限定为单镜像，扩展为镜像（Image）与 Helm Chart 两类制品。
- 原因：满足用户以不同交付形态部署 Agent 的需求，降低多工作负载 Agent 的接入成本。
- 影响：需求文档新增 AgentArtifact 概念；后续设计需覆盖 Chart 版本与 values 覆盖能力。

### ADR-049: Application 关系模型从“1 个 Agent + N 个组件”调整为“M 个 Agent + N 个组件”

- 决策：Application 由多个 Agent 实例与多个组件实例组成，Agent 实例与 Application 关系改为 1:N。
- 原因：当前业务模型明确要求 App 代表一组 Agent 实例与组件实例的组合编排。
- 影响：关系持久化新增 `ApplicationAgentRef`；Application 状态改为聚合计算。
- 替代关系：替代 ADR-037 中 Application 与 Agent 实例一对一关系。

### ADR-050: Agent 实例保持独立资源，但在同一时刻仅允许归属一个 Application

- 决策：沿用 Agent 实例独立 CRUD 能力，同时增加“单实例单归属”约束，避免跨应用共享引入生命周期冲突。
- 原因：兼顾独立运维与应用边界清晰，控制删除与变更的语义复杂度。
- 影响：应用创建/更新需校验 Agent 实例占用状态；关系表需强约束唯一归属。

### ADR-051: 级联删除语义适配多 Agent 模型

- 决策：Application 级联删除时默认删除其关联 Agent 实例；组件实例仍遵循“共享引用阻断删除（409）”策略。
- 原因：Agent 实例在当前模型下属于应用编排成员，生命周期应跟随应用；组件存在跨应用复用需保护。
- 影响：删除流程需先做组件引用计数，再执行 Agent 及派生资源回收。

### ADR-052: 设计输入待决项 O-001 关闭，统一使用 `resource_version` 并发控制

- 决策：更新与删除接口并发控制字段固定为 `resource_version`，不采用 `If-Match`。
- 原因：减少并发控制实现分叉，降低前后端交互复杂度。
- 影响：API 契约与资源模型统一以 `resource_version` 为冲突检测依据。

### ADR-053: 设计输入待决项 O-002 关闭，Secret 使用 K8S 默认机制

- 决策：当前阶段不引入外部 KMS 或数据库加密扩展，Secret 加密机制采用 K8S 默认策略。
- 原因：先保证部署链路稳定与交付效率，避免引入额外密钥基础设施复杂度。
- 影响：需求文档移除额外 KMS 待决项；后续如需增强加密能力单独立项。

### ADR-054: 设计输入待决项 O-003 关闭，剔除 Ingress 相关范围

- 决策：Agent 运行平台不考虑 Ingress 能力，需求与设计输入中移除 Ingress 相关内容。
- 原因：Ingress 属于集群网络入口治理范畴，不在当前部署模块职责边界内。
- 影响：组件与应用部署侧仅保留 Service 暴露策略（ClusterIP/NodePort）。

### ADR-055: 设计输入待决项 O-004 关闭，解绑恢复窗口默认 24h

- 决策：工作空间与集群解绑的可恢复窗口默认时长固定为 24 小时。
- 原因：平衡误操作恢复能力与资源回收效率。
- 影响：解绑流程实现与运维手册按 24 小时窗口执行。

### ADR-056: 设计输入待决项 O-005 关闭，Query 不做聚合且操作保持原子

- 决策：Query 接口不做跨资源状态聚合；资源操作语义保持原子性。
- 原因：降低查询路径复杂度与不一致风险，提升接口行为可预期性。
- 影响：Application 查询展示成员资源独立状态，不在查询阶段重算聚合状态。

### ADR-057: 设计输入待决项 O-006 关闭，Chart values 以前端部署参数为最高优先级

- 决策：Helm Chart 部署时，前端提交参数作为最高优先级 values，效果等价 `helm install -f values.yaml`。
- 原因：确保用户显式输入能稳定覆盖默认配置，符合 Helm 使用习惯。
- 影响：Chart 参数合并策略在实现侧固定，避免多来源 values 冲突歧义。

### ADR-058: 设计输入待决项 O-007 关闭，依赖镜像接口校验 + OCI 默认策略

- 决策：Chart 依赖通过接口检查工作空间关联镜像仓库内依赖镜像是否存在；OCI Chart 源校验采用默认策略。
- 原因：先保障依赖可用性校验闭环，同时避免过早引入定制校验复杂度。
- 影响：增加依赖检查接口调用路径；OCI 源校验维持默认实现。
- 替代关系：与 ADR-042 共同构成“待决项已关闭”最终状态。

### ADR-059: 平台内置高可用部署模板收敛为 Helm-only

- 决策：平台内置组件的高可用部署模板仅支持 Helm 形式，不再支持 Kustomize 与原生 YAML 路线。
- 原因：统一模板实现与运维路径，减少多渲染路线并行维护带来的复杂度与一致性风险。
- 影响：`requirements.md` 的 R-DSP-002 已更新为 Helm-only；
  `ComponentTemplate` 备注同步收敛为 Helm 模板元数据。
- 替代关系：替代 ADR-020。

### ADR-060: 资源模型扩展为 5 类一级资源并明确高/低代码应用边界

- 决策：部署模块一级资源固定为 5 类：数据服务组件、低代码平台、DevBox、网关、高代码应用；高代码与低代码应用语义分离。
- 原因：补充需求已将平台能力边界扩展到“应用 + 平台 + 开发环境 + 网关”全链路，原先仅围绕组件/Agent 的模型已不足。
- 影响：`requirements.md` 重构为按 5 类资源拆解需求，并新增高/低代码应用定义与边界说明。

### ADR-061: 全资源统一抽象与统一部署行为复用

- 决策：5 类资源统一落到“模板/制品 + 实例 + 关系 + 发布记录”抽象，并复用同一 CUD 执行链路。
- 原因：所有资源最终都部署为 K8S Pod，若逐资源独立实现将导致流程割裂、扩展成本高。
- 影响：后续设计必须提供可扩展适配器（默认 Helm），新增资源类型时不得重写任务、鉴权、审计主流程。

### ADR-062: 数据服务组件拆分为共享实例与嵌入式实例两种可见性域

- 决策：数据服务实例增加 `shared/embedded` 可见性语义；低代码平台依赖组件和高代码 Chart 自带组件统一归类为 `embedded`。
- 原因：补充需求明确这些实例不能与用户手工创建的共享组件混合展示和管理。
- 影响：列表查询、选择器、权限校验和删除策略均需按可见性域分离；`embedded` 仅随宿主资源生命周期管理。

### ADR-063: 低代码平台导入与开通策略固定为 Helm-only + 超管开通

- 决策：内置 Dify/FastGPT 与外部导入 Coze Studio/n8n/外部 Dify/FastGPT 统一走 Helm Chart；工作空间开通动作由超级管理员执行。
- 原因：补充需求已限定导入格式与开通角色，需在需求层固化防止实现分叉。
- 影响：低代码平台实例成为独立资源，Aether 管理平台实例生命周期，不管理其内部应用对象 CRUD。

### ADR-064: 网关实例采用 workspace+cluster 单例约束

- 决策：网关（Higress）在同一 `workspace + cluster(namespace)` 下最多 1 个实例。
- 原因：补充需求明确“一个工作空间内只能有一个网关”并与 namespace 映射绑定。
- 影响：网关创建需唯一性校验；应用发布前需校验网关可用性。

### ADR-065: DevBox 产物门禁规则定稿为“仅已发布镜像可建高代码应用”

- 决策：DevBox 内开发产物必须先经过平台发布流程生成发布镜像，未发布镜像不得用于高代码应用创建。
- 原因：补充需求强调 DevBox 的交付闭环，要求发布行为成为高代码应用创建前置条件。
- 影响：新增 DevBox 发布记录实体与校验链路；高代码应用创建需检查发布状态。

### ADR-066: 高代码应用发布必须沉淀可导出的 Helm Chart 版本

- 决策：高代码应用每次发布都必须生成标准化 Helm Chart，版本化存入工作空间关联镜像仓库，并支持下载导出。
- 原因：补充需求要求平台输出可迁移的应用部署包，便于跨环境导入。
- 影响：新增发布包实体与接口；发布流程从“部署成功”扩展为“部署 + 打包 + 入库 + 可下载”完整闭环。

### ADR-067: 领域模型分层定稿为“5 类一级资源 + 支撑资源”，Pod 不作为一级领域实体

- 决策：保持 5 类一级资源口径不变（数据服务组件、低代码平台、
  DevBox、网关、高代码应用）；`AgentInstance`、`HighCodeArtifact`、
  `DevBoxPublishRecord`、`HighCodeReleaseChart`
  与关系引用对象归类为支撑资源。Pod 仅作为运行时观测对象，
  不作为控制面一级领域实体。
- 原因：虽然所有类型最终运行在 K8S Pod 上，但直接按 Pod 建模会丢失权限、关系、可见性和发布语义，且与 ADR-060 的业务边界定义冲突。
- 影响：`requirements.md` 在“资源分层与关键概念”
  “统一资源抽象与可扩展部署行为”“领域模型与唯一性约束”
  “资源状态机”章节补充分层口径；后续 ERD/API 设计需遵循该分层，
  禁止将支撑资源口径扩张为“第 6 类一级资源”。

### ADR-068: 需求覆盖审计采用“已覆盖/待补充/不适用”三态并逐条核对

- 决策：在 `design.md` 的 `14.4` 章节建立逐条需求 ID 覆盖核对表，统一使用 `已覆盖`、`待补充`、`不适用` 三态。
- 原因：在分阶段编写设计文档（T1~T6）时，需要可追踪地表达“已落地”和“待补齐”的边界，避免误判完成度。
- 影响：每次需求或设计变更后必须同步更新 `14.4` 的映射章节与状态；未更新覆盖表不得关闭对应任务。

### ADR-069: 需求变更进入设计采用“决策先行 + 三文档联动 + 版本规则”流程

- 决策：需求变更必须按
  `decision.md -> requirements.md -> design.md` 顺序闭环，并在
  `design.md` 的 `15.2` 固化版本升级规则（MAJOR/MINOR/PATCH）。
- 原因：需求持续演进场景下，若缺少统一入口和版本规则，容易出现“需求、决策、设计”三者不一致。
- 影响：实现前必须完成 ADR 记录、需求文本更新、设计覆盖状态更新与任务状态同步；版本号用于评审与交付基线追踪。

### ADR-070: 统一部署扩展点固定为五段式流水线

- 决策：将统一执行链路的可扩展点固定为
  `SchemaValidator -> TemplateResolver -> DeployAdapter -> StatusProjector -> AuditEmitter`
  五段式流水线。
- 原因：满足“新增资源类型不重写任务、鉴权、审计主流程”的约束，避免不同资源各自拼装执行链路。
- 影响：后续新增资源类型仅允许新增 schema、模板解析与适配器实现；任务框架与状态回写机制保持复用。

### ADR-071: 软删模型下的唯一约束采用部分唯一索引

- 决策：所有业务唯一约束在数据库层采用 `deleted_at is null` 的部分唯一索引实现。
- 原因：既要保留软删审计历史，又要允许同名资源在删除后重建；普通唯一索引无法同时满足。
- 影响：数据库迁移需分阶段执行（先回填、后建索引）；读写路径默认追加 `deleted_at is null` 过滤。

### ADR-072: 工作空间解绑生命周期状态显式持久化到绑定模型

- 决策：为 `WorkspaceClusterBinding` 显式持久化解绑生命周期状态：`Active/Frozen/Validating/RecoveryWindow/Reclaiming/ReclaimFailed/Unbound`。
- 原因：解绑流程跨越异步任务与 24h 恢复窗口，仅靠任务日志无法稳定表达当前阶段和恢复入口。
- 影响：解绑、恢复、回收接口均以绑定状态机驱动；审计与告警可按状态直接检索与追踪。

## 2026-02-23

### ADR-073: 外部资源 API 路径统一为 workspace+cluster 分层

- 决策：对外资源接口统一采用
  `/api/v1/workspaces/{workspace_id}/clusters/{cluster_id}/...`
  路径模式；CUD 统一异步，Query 保持同步。
- 原因：在接口层显式固定租户与部署边界，减少跨边界误用；同时与需求中的异步变更模型保持一致。
- 影响：OpenAPI 分组、鉴权中间件和 handler 参数绑定均按该路径结构落地；禁止从请求体覆盖 `workspace_id/cluster_id`。

### ADR-074: 幂等键采用“作用域哈希 + 请求指纹”双重校验

- 决策：`Idempotency-Key` 去重不做全局唯一，改为
  `actor_id + workspace_id + cluster_id + method + route_template + idempotency_key`
  组成作用域；同作用域若请求指纹不一致返回
  `409 IDEMPOTENCY_PAYLOAD_MISMATCH`。
- 原因：避免不同接口或不同资源误命中同一幂等记录，同时阻断同键改参导致的语义歧义。
- 影响：任务生产者需维护幂等作用域与请求指纹；API 错误模型新增 `IDEMPOTENCY_PAYLOAD_MISMATCH`。

### ADR-075: 任务取消采用协作式中断，不强制终止执行进程

- 决策：任务取消语义定义为：`Pending` 可直接取消，`Running` 仅设置 `cancel_requested` 标记，由执行器在阶段边界检查并协作中断。
- 原因：K8S/OCI 操作多为外部副作用，强杀执行进程会放大中间态与资源漂移风险。
- 影响：任务模型新增取消标记字段；适配器需在关键步骤间插入取消检查点，并记录取消审计轨迹。

### ADR-076: Chart values 合并策略定稿为四层优先级并在渲染前统一校验

- 决策：Chart values 按“接口 `values_override` > 资源 `spec`
  映射值 > 工作空间/集群策略默认值 > 模板默认值”四层顺序合并；
  渲染前统一执行 schema 校验，失败即返回 `422`。
- 原因：多来源 values 并存时需要固定覆盖顺序，避免不同资源适配器出现参数冲突与行为漂移。
- 影响：`DeployAdapter` 必须复用统一的 values merge 与校验组件，不允许资源私有实现分叉。
- 替代关系：细化 ADR-057 的“前端参数最高优先级”原则为可编码的完整合并顺序。

### ADR-077: Helm 执行策略定稿为“原子部署优先 + 漂移修复串行”

- 决策：部署/升级默认启用 `helm upgrade --install --atomic --wait --timeout`；失败先自动回滚到最近成功 revision。漂移修复任务与业务变更任务共享同一串行键执行。
- 原因：统一失败处理可以降低中间态风险；漂移修复与人工变更并发执行会导致状态回写互相覆盖。
- 影响：任务执行器在失败分支必须先尝试回滚再落库终态；reconcile 任务进入同资源串行队列。

### ADR-078: Secret 回滚语义定稿为“生成新版本并切换指针”

- 决策：Secret 回滚不直接回拨当前版本指针到历史版本，而是读取目标历史版本内容，生成新版本 `v(n+1)` 并切换当前指针。
- 原因：直接回拨会破坏版本时间线与审计连续性，不利于追溯“谁在何时触发了回滚”。
- 影响：`SecretVersion` 生命周期新增 `Retained/Archived` 语义；回滚流程与更新流程统一使用“新版本写入 + 指针切换”机制。

### ADR-079: 指标标签采用低基数白名单，高基数字段仅保留在日志与追踪

- 决策：Prometheus 指标标签仅允许低基数字段（如 `task_type`、
  `resource_kind`、`result`）；`workspace_id/resource_id/task_id`
  不进入指标标签。
- 原因：高基数标签会造成时序膨胀和查询性能下降，影响生产可观测性稳定性。
- 影响：监控面聚合能力与告警规则统一按低基数标签实现；细粒度定位依赖 `request_id/task_id` 关联日志与 trace。

### ADR-080: NFR 验收采用“分层压测 + 稳定性演练”双门禁

- 决策：NFR 验收必须同时通过接口压测、队列压测、端到端压测与 72 小时稳定性演练，任一失败则不得进入发布。
- 原因：仅做单点性能测试无法覆盖异步队列、外部依赖抖动和长时间状态投影延迟等真实生产风险。
- 影响：`design.md` 第 13 章落地为可执行验收方案；发布流程新增 NFR 门禁检查项并绑定 SRE/后端联合签署。

### ADR-081: T6 阶段验收矩阵采用“需求 ID 分组 + 用例 ID + 证据产物”结构

- 决策：`design.md` 的 `14.1` 采用统一矩阵结构：`需求 ID -> 测试场景(用例 ID) -> 验收标准 -> 证据产物`，并要求每个需求簇至少一条可执行用例。
- 原因：仅有章节映射不足以支撑交付验收；需要把需求直接绑定到可执行测试和可归档证据。
- 影响：测试执行与发布评审必须按该矩阵逐项勾检；若缺少证据产物，不得标记需求为验收通过。

### ADR-082: 发布准入固定为四道 Gate，回滚触发绑定 NFR 阈值与安全事件

- 决策：发布流程固定为 Gate-1（需求设计一致性）、
  Gate-2（P0 用例与缺陷）、Gate-3（NFR 指标）、
  Gate-4（变更安全与回滚演练）；回滚触发条件绑定
  `design.md` 第 13 章阈值与 P0 安全事件。
- 原因：将“可发布”从经验判断转为可审计、可重复执行的门禁机制，降低上线风险。
- 影响：发布前必须完成 Gate 记录；值班流程需支持“一键进入回滚 Runbook”并归档回滚证据。

### ADR-083: 开放问题采用“Owner + 截止日期 + 关闭条件”强约束管理

- 决策：`design.md` 的开放问题清单必须为每条问题定义 `Owner`、`目标关闭日期` 与 `关闭条件`，并在周会进行状态复核。
- 原因：待决策项若只做文本记录，容易长期悬置并在实现阶段形成隐性风险。
- 影响：延期需显式说明原因和新日期；问题关闭后需同步回写 `decision.md` 与 `design.md` 相关章节。

### ADR-084: T7 契约一致性收口采用“必选端点补齐 + canonical path 单口径 + 覆盖回标”三联动作

- 决策：T7 阶段统一采用 `/api/v1/workspaces/{workspace_id}/clusters/{cluster_id}/...`

### ADR-088: T11 仓库绑定回滚契约定稿为“显式内部接口 + 任务化回滚 + 证据闭环”

    `confirmation_token`。

    版本不存在返回 `404 BINDING_VERSION_NOT_FOUND`；

- 替代关系：对 ADR-036（解绑恢复窗口）与 ADR-085（契约冲突修订）的“回滚可执行性”补充内部接口与任务编排层实现口径，以本 ADR 为当前生效口径。

  - 验收新增 `TC-LCP-04-A~D`，用于验证四字段“可写、可读、可审计、可回滚”。
- 替代关系：对 ADR-086 中 `R-LCP-008` 相关结论保持不变，并补齐其未覆盖的
  `R-LCP-007` 模型层缺口；以本 ADR 作为低代码四字段闭环的当前生效口径。

### ADR-092: T16 语义补遗定稿为“导入可调用 + 删除可执行 + 凭证链路闭环 + 回标防回归”

- 决策：
  - `R-LCP-003` 的低代码 Helm 导入能力固定为显式异步端点
    `POST /api/v1/workspaces/{workspace_id}/clusters/{cluster_id}/lowcodes/imports`，
    仅允许 `super_admin` 调用，并强制最小请求字段
    `instance_name/platform_type/chart_ref/chart_version/entry_url/admin_account_ref/version`。
  - `R-LCP-004` 的 LowCode CRUD 明确补齐
    `DELETE /api/v1/workspaces/{workspace_id}/clusters/{cluster_id}/lowcodes/{lowcode_id}`；
    删除请求必须携带 `Idempotency-Key + resource_version`，返回 `202 + task_id`，
    并定义 embedded 回收失败进入补偿任务。
  - `R-ENV-007` 的仓库凭证消费语义固定为“image/chart push/pull 同口径”：
    DevBox 镜像发布链路必须执行
    `registry_login -> image_push -> digest_verify`，
    并写入 `registry_binding_version/image_ref/digest/failure_reason/retry_count`。
  - 需求覆盖回标固定执行“先回退为待补充，再恢复为已覆盖并附证据章节”的闭环流程，
    防止“ID 已覆盖但正文契约缺失”复发。
- 原因：
  - T15 完成后仍存在三类剩余缺口（GAP-20~22）：
    导入能力仅有描述无可调用契约、LowCode DELETE 未显式落盘、镜像 push 凭证链路缺执行与审计细节。
  - 若继续维持“覆盖表已覆盖但正文不完整”，实现阶段会出现接口不可落地、验收证据不可执行的风险。
- 影响：
  - `design.md` 已同步修订 `4.4`、`4.5`、`7.3`、`7.5`、`8.2.1`、`8.5`、`9.3`、
    `10.5`、`14.1.8`、`14.4.13`、`15.2`。
  - `task_design.md` 的 T16 与 T16-01~T16-04 可标记完成，并关闭 GAP-20~GAP-22。
  - 验收新增 `TC-LCP-05-A/B`、`TC-DBX-03`，用于验证导入契约、DELETE 契约与凭证链路闭环。
- 替代关系：对 ADR-089（接口缺项与权限补齐）和 ADR-091（低代码字段闭环）做二次覆盖复核收口，以本 ADR 作为 T16 生效口径。

### ADR-093: T17 错误码词典定稿为“403 双码互斥 + 409 引用冲突单码 + 兼容别名限窗退场”

- 决策：
  - 403 权限拒绝固定为两个 canonical 业务码并互斥触发：
    `FORBIDDEN_WORKSPACE`（无工作空间权限或跨边界调用）、
    `FORBIDDEN_ACTION`（动作越权，含非超管执行超管动作、租户调用 internal 接口）。
  - 409 shared 引用冲突固定为单一 canonical 业务码
    `REFERENCE_CONFLICT`，统一用于删除决策表、OpenAPI 与验收断言。
  - 旧码兼容策略固定为限窗退场：
    `FORBIDDEN`、`SHARED_RESOURCE_IN_USE` 仅在 `v1.11~v1.12`
    做响应解析兼容与日志归并，`v1.13` 起禁止继续返回。
  - 兼容窗口期间必须产出告警与监控：
    `error_code_legacy_alias_total{alias=*}`，用于发布门禁检查。
- 原因：
  - T16 后仍存在同类场景多业务码并存（`FORBIDDEN/FORBIDDEN_WORKSPACE`、
    `REFERENCE_CONFLICT/SHARED_RESOURCE_IN_USE`），导致实现方无法建立稳定错误处理分支，
    且验收断言与 OpenAPI 存在双口径风险。
  - `R-OPS` 与 `R-HCA-012` 需要“单场景单错误码”以支持自动化重试、告警聚合与跨章节一致性审计。
- 影响：
  - `design.md` 已同步修订 `7.2`、`7.5`、`10.2`、`11.4`、`14.1.9`、`14.4.14`、`15.2`。
  - `task_design.md` 的 T17 与 T17-01~T17-03 可标记完成，并关闭 GAP-23。
  - 后续新增接口若涉及 403/409，必须先通过“错误码词典一致性复核”检查点。
- 替代关系：对 ADR-089（权限口径）与 ADR-044（级联冲突 409 语义）补充实现级错误码命名约束，以本 ADR 作为 T17 生效口径。

### ADR-094: T18 幂等字段命名定稿为“canonical 全称 + 缩写别名限域兼容”

- 决策：
  - 幂等字段 canonical 命名固定为：
    `idempotency_scope`、`idempotency_key`、`request_fingerprint`、
    `idempotency_expire_at`。
  - `async_tasks` 模型、唯一约束、TTL 索引、流程与验收断言全部统一使用 canonical 命名；
    禁止在正文继续使用 `idem_scope/idem_key/fingerprint/expire_at`。
  - 历史缩写别名仅允许出现在“迁移说明/历史数据释义”场景，不得进入主模型表、DDL 命名、OpenAPI 示例与验收断言。
  - 缩写别名退场策略固定为：
    `v1.13.0` 起仅保留内部日志解析映射，
    文档与实现对外输出统一为 canonical 字段名。
- 原因：
  - T17 后仍存在 `5.2.3` 使用 `idem_*`、其余章节使用 `idempotency_*` 的双口径问题，影响模型实现、索引命名和验收断言一致性。
  - 幂等字段属于高频排障与自动化校验字段，命名不一致会放大跨模块沟通与实现歧义成本。
- 影响：
  - `design.md` 已同步修订 `2.6`、`5.2.3`、`5.4`、`8.1`、`8.4`、
    `14.1.10`、`14.4.15`、`15.2`。
  - `task_design.md` 的 T18 与 T18-01~T18-04 可标记完成，并关闭 GAP-24。
  - 后续涉及幂等字段的改动必须执行“命名一致性复核”，否则不得回标 `已覆盖`。
- 替代关系：对 ADR-085/ADR-087 中幂等口径结论补充“字段命名治理”实现约束，以本 ADR 作为 T18 生效口径。

### ADR-095: T19 执行基础设施与主数据边界定稿为“PG 队列与锁 + Outbox Relay + JetStream + 表级写入责任矩阵”

- 决策：
  - 任务队列定稿为 PostgreSQL `async_tasks` + lane 分层（`lane_reclaim_high/lane_cud/lane_projection`），消费采用 `FOR UPDATE SKIP LOCKED`。
  - 分布式锁定稿为 PostgreSQL advisory lock，并引入 `fencing_token` 防止过期持锁写入覆盖新执行。
  - 事件链路定稿为 PostgreSQL Outbox + 轮询 Relay + NATS JetStream（至少一次投递，`event_id` 去重，DLQ 重放）。
  - 主数据读写边界定稿：
    - `workspaces/managed_clusters` 由平台管理面独占写入，Aether 只读；
    - `workspace_*_bindings` 由 Aether internal API 作为唯一写入入口；
    - 禁止跨服务直连写入对方 owner 表。
  - 越界写入定稿为“DB 权限拒绝 + 触发器审计 + P1 告警”三层防护。
- 原因：
  - T18 完成后仍存在 GAP-25/GAP-26：执行底座虽有流程描述但缺技术定稿，主数据边界缺实现级约束，存在落地歧义与越界风险。
  - 当前吞吐目标（峰值 `>=2.4 task/s`）下，PG 队列方案可满足性能且能保证任务与状态同事务一致性。
  - 绑定变更链路需要可靠事件分发与可重放能力，Outbox + JetStream 组合可在复杂度与可靠性间取得平衡。
- 影响：
  - `design.md` 已同步修订 `3.2`、`4.1`、`5.7`、`7.6`、`8.1~8.9`、`11.2`、`12.2`、`13.1`、`14.1.11`、`14.4.16`、`15.2`、`15.3`。
  - `task_design.md` 的 T19 与 T19-01~T19-05 可标记完成，并关闭 GAP-25/GAP-26。
  - OQ-04（任务优先级策略）关闭，后续如调整 lane 配额需走 `15.2` 变更流程并回标。
- 替代关系：补充并收敛 ADR-088（绑定回滚编排）与 ADR-092（执行链路补遗）中的执行底座未定项，以本 ADR 作为 T19 生效口径。
