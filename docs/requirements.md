# Aether 模块需求文档

## 1. 文档目标

本文档定义 Agent 平台中 Aether 部署模块的完整需求基线，覆盖：

- 六类资源的统一 CRUD（按原需求记法，CUD 异步、Query 同步）
- 统一接口入参与返回值约束
- 多角色、多工作空间、多集群场景下的资源作用域与权限边界
- 可扩展到新资源类型的通用部署抽象
- 与 Aether 解耦的非目标边界（如 Helm Chart 产物生成与仓库版本管理）

## 2. 背景与范围

### 2.1 业务背景

Agent 平台需同时支持：

- 高代码 Agent 应用：DevBox 发布镜像或用户上传镜像/Helm Chart 创建
- 低代码 Agent 应用：平台内置 Dify/FastGPT 或用户上传 Helm Chart 的 Coze Studio/n8n

所有资源底层均通过 Helm 在 Kubernetes 部署。

### 2.2 Aether 责任范围（In Scope）

- 提供通用、复用的资源 CUD/Query 能力
- 统一资源生命周期管理（创建、更新、删除、查询）
- 统一异步任务编排与状态回传
- 基于工作空间/集群/命名空间的部署落地
- 为上层模块提供可复用接口函数（非对外 OpenAPI）

### 2.3 非目标（Out of Scope）

- 不负责高代码应用 Helm Chart 产物生成
- 不负责 Helm Chart 版本发布到镜像仓库
- 不负责低代码平台内部业务功能（如 Dify/FastGPT 的工作流细节）
- 不负责平台账号登录认证流程本身

## 3. 术语与对象

- 工作空间（Workspace）：资源管理与权限作用域的核心边界
- 纳管集群（Managed Cluster）：被平台接入的 Kubernetes 集群
- 命名空间（Namespace）：工作空间在关联集群中的同名 namespace
- 纳管镜像仓库（Managed Registry）：被平台接入的 Harbor 仓库
- 资源类型（Resource Type）：Aether 管理的部署对象类型
- 资源实例（Resource Instance）：某资源类型的一次实例化部署
- 异步任务（Async Task）：CUD 动作的执行单元
- 内置依赖数据服务（Embedded Data Service）：低代码平台/记忆体/某些 Helm 应用自带且仅供自身使用的数据服务实例

## 4. 参与角色与权限要求

### 4.1 角色定义

- 超级管理员
- 空间管理员
- 普通用户

### 4.2 权限规则

#### FR-ACL-001 超级管理员权限

超级管理员可操作全部资源，至少包括：

- 纳管 K8S 集群 CRUD
- 纳管镜像仓库（Harbor）CRUD
- 工作空间 CRUD
- 工作空间关联集群 CRUD
- 工作空间关联镜像仓库 CRUD
- 工作空间网关实例 CRUD
- 工作空间空间管理员配置
- 空间管理员与普通用户全部能力

#### FR-ACL-002 空间管理员权限

空间管理员可在授权工作空间内：

- 开通低代码平台
- 关联与管理普通用户（CRUD）
- 执行普通用户全部能力

#### FR-ACL-003 普通用户权限

普通用户可在被关联工作空间内：

- 创建数据服务组件实例
- 使用 DevBox（CRUD）
- 创建记忆体实例
- 创建高代码应用
- 进入工作空间内低代码平台并操作

#### FR-ACL-004 工作空间内无细粒度隔离

同一工作空间内用户可操作该空间内资源，不做按“资源创建者”维度的二级隔离。

## 5. 资源类型与业务规则

### 5.1 资源类型清单

#### FR-RT-001 Aether 必须支持以下六类资源

- 数据服务组件：PostgreSQL、Redis、Milvus、Weaviate、pgvector、NebulaGraph、MongoDB
- 低代码平台：Dify、FastGPT、Coze Studio、n8n
- DevBox
- 网关（Higress）
- 高代码应用（openclaw、zeroclaw，以及用户自定义镜像/Chart）
- 记忆体（mem0）

### 5.2 资源级约束

#### FR-RT-002 数据服务组件约束

- 数据服务组件模板由平台内置（Helm Chart）
- 高代码应用可绑定用户预先创建的数据服务组件
- 低代码平台/记忆体部署时创建的内置依赖数据服务，不与“用户独立创建的数据服务组件”混合展示与统一管理

#### FR-RT-003 低代码平台约束

- 平台内置 Dify/FastGPT 模板，且模板形态为 Helm Chart
- 支持超级管理员导入 Coze Studio、n8n 或外部 Dify/FastGPT，导入格式仅 Helm Chart
- 同一工作空间内内置低代码平台每种最多一个实例
- 上述“每种最多一个实例”仅约束平台内置低代码平台实例，不约束用户导入的同类型外部 Chart 实例
- 部署时依赖数据服务随平台一并创建

#### FR-RT-004 DevBox 约束

- DevBox 基于平台内置高代码模板（Helm Chart）创建开发容器
- 仅发布后的镜像可用于创建高代码应用

#### FR-RT-005 网关约束

- 网关模板为平台内置 Helm Chart
- 同一工作空间在同一集群（对应 namespace）内最多一个网关实例

#### FR-RT-006 高代码应用约束

- 支持三种来源创建：DevBox 发布镜像、用户上传镜像、用户上传 Helm Chart
- 镜像方式：1 个 Agent 实例 + 1..N 个预创建数据服务组件
- Helm Chart 方式：1..N 个 Agent 实例 + 1..N 个预创建数据服务组件
- Helm Chart 自带的数据服务组件仅对该应用可见，不纳入通用数据服务组件列表统一管理

#### FR-RT-007 记忆体约束

- 模板形态为平台内置 Helm Chart
- 一个记忆体实例由 1..N 记忆体业务实例 + 1..N 不同数据服务组件实例构成
- 记忆体依赖数据服务随实例部署创建，且仅供该记忆体实例使用

## 6. 工作空间与多集群映射

### FR-WS-001 工作空间与集群命名空间映射

- 每个工作空间可关联 1..N 纳管集群
- 每个关联集群中必须创建与工作空间同名 namespace
- 若关联多个集群，必须在每个集群均创建同名 namespace

### FR-WS-002 工作空间与镜像仓库关联

- 每个工作空间可关联 0..N 纳管镜像仓库
- Aether 需保存资源部署与镜像/Chart 来源引用关系（不负责仓库治理业务）

### FR-WS-003 部署位置必选

- 资源创建请求必须显式传入部署位置，部署位置即目标 K8S 集群
- 部署位置为必选参数，未传入时请求必须被拒绝并返回参数错误

## 7. Aether 通用能力需求

### 7.1 通用抽象与扩展机制

#### FR-CORE-001 统一部署抽象

Aether 必须提供统一资源部署抽象，至少包括：

- 资源类型注册机制（类型标识、模板来源、校验规则、默认值规则）
- 通用 Helm 生命周期执行器（install/upgrade/uninstall）
- 通用状态采集器（任务状态、资源状态、部署修订）

#### FR-CORE-002 可扩展性

新增资源类型时，不允许复制粘贴现有业务流程；应通过“类型注册 + 配置化规则 + 复用执行器”完成接入。

#### FR-CORE-003 统一资源标识

所有资源实例必须具有全局唯一 `resource_id`，并包含 `workspace_id`、`cluster_id`、`namespace`、`resource_type`。

### 7.2 接口函数契约

#### FR-API-001 Aether 仅提供代码级接口

Aether 不直接暴露外部 OpenAPI，仅提供可供其他模块调用的 Go 接口。

#### FR-API-002 统一 CUD 入参

CUD 接口必须复用统一请求结构，最小字段包括：

- `request_id`（幂等请求号）
- `operator`（操作者身份与角色信息）
- `workspace_id`
- `target`（resource_type、resource_id，可选）
- `cluster_id`（目标部署集群，必填）
- `spec`（资源声明配置，含 Helm values/依赖引用）
- `labels` / `annotations`（可选）

#### FR-API-003 统一 CUD 返回

CUD 接口必须同步返回受理结果，不阻塞部署完成，最小字段包括：

- `task_id`
- `resource_id`（创建场景可先分配）
- `accepted_at`
- `status`（固定为 `ACCEPTED`）

#### FR-API-004 Query 接口能力

Query 接口必须同步返回，并支持：

- 分页：`page`, `page_size`
- 过滤：按 `workspace_id`、`resource_type`、`name`、`status`、`creator`、时间区间等
- 排序：支持通过请求参数传入排序字段与升降序；Aether 仅提供通用排序能力，不内置固定排序字段白名单与默认排序规则

#### FR-API-005 统一错误模型

接口错误必须采用统一错误码体系，至少区分：

- 参数错误
- 权限错误
- 资源不存在
- 资源冲突
- 依赖缺失
- 异步任务系统不可用
- 内部错误

### 7.3 异步任务模型

#### FR-TASK-001 CUD 全异步

Create/Update/Delete 均以异步任务执行，采用生产者-消费者模型。

#### FR-TASK-002 技术实现约束

- 任务队列必须使用 Redis
- 分布式锁必须使用 Redis
- 任务元数据与资源元数据持久化必须使用 PostgreSQL

#### FR-TASK-003 任务状态机

任务至少包含以下状态：

- `PENDING`
- `RUNNING`
- `SUCCEEDED`
- `FAILED`
- `CANCELLED`（若支持取消）

#### FR-TASK-004 幂等与重试

- 基于 `request_id + workspace_id + action + resource_key` 实现幂等去重
- 对可恢复错误支持重试策略（次数、退避可配置）
- 重试过程必须保证同一资源并发互斥（Redis 分布式锁）

#### FR-TASK-005 可观测性

任务需记录：

- 提交时间、开始时间、结束时间
- 当前步骤与失败原因
- 关联资源与 Helm release 标识

### 7.4 查询与一致性

#### FR-Q-001 查询一致性要求

Query 返回“已持久化状态”，对异步执行中的资源需明确返回中间态（如 `CREATING`、`UPDATING`、`DELETING`）。

#### FR-Q-002 任务查询

必须提供任务查询接口函数（按 `task_id`、`resource_id`、分页）供上层轮询或展示。

## 8. 数据模型需求

### FR-DM-001 核心实体

至少需要以下实体（可拆分多表）：

- 工作空间映射信息（workspace-cluster-namespace）
- 资源实例主表
- 资源版本/修订表
- 资源依赖关系表
- 异步任务表
- 任务事件/日志表

### FR-DM-002 依赖关系表达

资源模型需支持以下依赖表达：

- 高代码应用 -> 用户创建数据服务组件
- 低代码平台 -> 内置依赖数据服务组件（隐藏）
- 记忆体 -> 内置依赖数据服务组件（隐藏）
- Helm 高代码应用 -> 自带依赖数据服务组件（隐藏）

### FR-DM-003 可见性标记

数据服务组件必须区分可见性类型，至少包括：

- `USER_MANAGED`（用户可见可管理）
- `EMBEDDED_PRIVATE`（仅归属父资源可见）

### FR-DM-004 删除级联规则

- 当父资源删除时，其关联的 `EMBEDDED_PRIVATE` 依赖必须强制级联删除
- 不允许对 `EMBEDDED_PRIVATE` 依赖采用“保留”策略

## 9. 非功能需求

### NFR-001 技术栈约束

- 后端语言：Go
- Web/API 框架：Gin
- 持久化数据库：PostgreSQL
- 队列与分布式锁：Redis
- 模板/部署包标准：Helm Chart

### NFR-002 可扩展性目标

新增资源类型不应修改通用任务编排核心逻辑，仅新增类型定义和规则。

### NFR-003 可靠性目标

- 异步任务在进程重启后可恢复
- 任务执行必须具备至少一次投递保障
- 对同一资源并发变更应避免竞态破坏

### NFR-004 可审计性目标

所有 CUD 行为需可追溯到操作者、请求参数摘要、执行结果与时间线。

## 10. 边界集成需求

### FR-INT-001 与上层模块集成方式

各业务模块（高代码应用、低代码平台、DevBox、网关等）负责：

- 接收自身 OpenAPI 请求
- 将业务请求转换为 Aether 统一入参
- 调用 Aether 通用接口函数

### FR-INT-002 与 Helm Chart 产物服务解耦

“生成高代码应用 Helm Chart 并发布版本到镜像仓库”的逻辑必须由独立模块实现，Aether 仅消费其产物引用，不承载该流程。

## 11. 验收标准

### AC-001 统一接口

六类资源均可通过同一套 CUD/Query 接口完成生命周期操作，无需为每类资源单独实现一套引擎。

### AC-002 异步执行

CUD 请求返回 `task_id`，并可通过任务查询接口观察状态流转直至完成。

### AC-003 查询能力

资源与任务查询均支持分页、过滤、排序，并在工作空间维度满足权限和可见性规则。

### AC-004 可见性隔离

内置/自带依赖数据服务组件不会出现在用户通用数据服务组件列表中，仅在父资源上下文可见。

### AC-005 约束校验

系统可正确拒绝违反约束的请求（例如同工作空间重复创建内置低代码平台实例、重复网关实例）。

### AC-006 解耦边界

高代码应用 Helm Chart 产物生成与仓库版本管理模块变更时，不影响 Aether 核心 CUD/Query 实现。
