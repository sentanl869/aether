# Aether 部署模块需求分析与拆解

## 背景与目标

Aether 是 AI Agent 运行平台的部署模块。
负责在平台已纳管的 K8S 集群上部署平台内置组件与用户 Agent 应用。
平台通过 kubeconfig 纳管集群，用户上传已构建的镜像到平台镜像仓库后，通过平台完成组件实例与 Agent 应用的部署。

平台内存在“工作空间”概念：一个工作空间可关联一个或多个 K8S 集群，并在每个关联集群中对应一个同名 namespace。

目标：

- 支持平台内置组件（如 milvus、mem0）的高可用部署与参数化配置。
- 支持用户 Agent 应用部署，并可关联已创建的组件实例。
- 支持用户使用自有 Agent 镜像时，直接选择已创建组件实例并一次完成应用创建。
- 支持对 K8S 上的 Agent 应用进行增删改查（CRUD）。
- 保存应用实例与组件关系，支撑后续应用查看与运维。
- 基于工作空间提供多集群边界管理与权限隔离。

## 需求范围

### In Scope

- 纳管 K8S 集群的部署与运维能力（通过 kubeconfig）。
- 工作空间与集群绑定管理，以及同名 namespace 映射规则。
- 内置组件模板化部署（参数化 + 高可用配置模板）。
- 组件实例创建、更新、删除、查询。
- 组件运行时深度运维（扩缩容、升级/回滚、滚动重启等）。
- Agent 实例独立部署与管理（CRUD）。
- Agent 应用部署（配置资源、镜像、规模等）。
- Agent 应用与组件实例关联并联合部署。
- 应用实例信息与组件关系的持久化。
- 镜像仓库凭证管理（用于推送/拉取镜像）。
- 工作空间隔离（资源边界、权限边界与跨边界关联限制）。
- 权限模型（超级管理员 / 用户）与工作空间级资源操作控制。
- 跨集群与跨 namespace 关联限制。
- Agent 实例、组件实例、Agent 应用的 CUD 采用生产者-消费者异步任务模型，Query 同步返回。

### Out of Scope

- 镜像仓库的搭建与镜像上传流程（由平台其他模块负责）。
- 服务网格等高级网络治理（当前未纳入设计）。

## 实现约束（技术栈）

- 后端实现语言必须使用 Go。
- Web/API 框架必须使用 Gin。
- 持久化数据库必须使用 PostgreSQL（Postgres）。

## 角色与使用场景

- 超级管理员：可操作全部工作空间、集群与资源；纳管集群，维护内置组件模板与镜像。
- 用户：仅可操作其关联工作空间内资源；可创建组件实例、部署 Agent 应用并关联组件、查看应用状态。

## 关键概念

- 工作空间（Workspace）：资源管理边界。一个工作空间可关联多个 K8S 集群；每个关联集群中均对应一个同名 namespace。
- 组件（Component）：平台内置中间件/基础能力，如 milvus、mem0。
- 组件实例（Component Instance）：根据模板与配置创建的具体部署。
- Agent 实例（Agent Instance）：用户镜像部署出的运行实例。
- 应用（Application/Agent App）：1 个 Agent 实例 + N 个组件实例的组合体；组件实例可被多个应用复用；保存关系用于查询与运维。

## 需求拆解

### 1. 集群纳管、工作空间与目标环境

1.1 使用 kubeconfig 纳管 K8S 集群。
1.2 集群资源可用于部署组件与 Agent 应用。
1.3 工作空间与集群建立关联关系后，系统需在每个关联集群中维护同名 namespace（不存在则创建）。
1.4 创建组件实例、Agent 实例、Agent 应用时，需先选择工作空间和目标集群；命名空间由工作空间映射决定，不允许手工选择其他 namespace。
1.5 镜像仓库凭证应可配置并下发至目标 namespace（用于拉取/推送）。
1.6 Secret 管理规则：

- Secret 必须保存在工作空间对应 namespace 内。
- 普通用户不可查看 Secret 明文，仅可在资源创建/更新时引用。
- 仅平台控制面服务账号可读取 Secret 明文并完成注入。
- Secret 更新支持版本化引用（新版本写入并切换引用，可用于回滚与审计）。
1.7 工作空间与集群解绑规则：
- 解绑流程采用“冻结 + 校验 + 可恢复窗口 + 回收”机制，不直接立即删除：
  - 冻结：禁止该工作空间在目标集群上新增部署与变更任务。
  - 校验：校验运行中任务与关键资源状态，生成影响面清单。
  - 可恢复窗口：窗口期内允许撤销解绑并恢复。
  - 回收：窗口结束后删除该工作空间在该集群对应 namespace 及其资源。
- 该操作仅允许超级管理员执行，且必须进行二次确认。

### 2. 平台内置组件部署

2.1 组件镜像存放在平台镜像仓库（平台部署后可用）。
2.2 提供组件高可用部署模板，技术路线优先级为：Helm > Kustomize > 原生 YAML。
2.3 组件实例创建时支持填写：

- 目标集群：仅可选择工作空间已关联集群。
- 资源规格：CPU、内存、GPU（如需）。
- 存储配置：选择 StorageClass（SC）与容量（Gi），不向用户暴露更底层存储实现细节。
- 规模：副本数量。
- 环境配置：数据库名、用户名、密码等。
- 可选的网络/服务暴露配置（ClusterIP/NodePort/Ingress）。
- Service 名称策略：
- 当组件在“应用创建时直接关联组件实例”场景下被引用时，Service 名称由平台自动生成。
- 当组件以“独立创建组件实例”方式创建且用户显式选择自定义时，允许用户自定义 Service 名称（需在同一 namespace 内唯一）。
2.4 组件实例部署、更新、删除、查询能力。
2.5 组件实例状态监控（部署成功/失败、运行态、事件摘要）。
2.6 组件运行时运维能力：
- 扩缩容（副本与资源规格）。
- 升级/回滚（镜像版本或模板版本）。
- 滚动重启与健康检查管理。

### 3. Agent 实例与 Agent 应用部署

3.1 用户上传 Agent 镜像后，可独立创建 Agent 实例。
3.2 Agent 实例部署时可配置：

- 目标集群：仅可选择工作空间已关联集群。
- 资源规格：CPU、内存。
- 存储配置：选择 StorageClass（SC）与容量（Gi），不向用户暴露更底层存储实现细节。
- 规模：副本数量。
- 必要环境变量（如模型配置、API Key）。
3.3 Agent 应用创建支持两种方式：
- 选择已创建 Agent 实例，再关联已创建组件实例并创建应用。
- 使用自有 Agent 镜像，在一次提交中完成 Agent 实例创建、组件关联与应用创建。
3.4 Application 与 Agent 实例关系为一对一；组件实例允许被多个 Application 复用。
3.5 组件关联限制：
- 仅允许关联同一工作空间、同一集群（同一 namespace）下的组件实例。
- 不允许跨 K8S 集群关联组件实例。
3.6 部署时自动生成：
- Deployment/StatefulSet、Service、ConfigMap/Secret（如需）。
- 与组件实例的连接信息注入（环境变量或配置文件）。
  - 连接信息包含用户名、密码、地址等；敏感信息统一通过 Secret 管理。

### 4. 应用关系与持久化

4.1 “应用”作为组合体：1 个 Agent 实例 + 组件实例关系。
4.2 应用创建后保存：

- 应用基本信息（名称、描述、创建者、工作空间、集群、命名空间）。
- Agent 部署参数与镜像信息。
- 关联的组件实例列表及引用方式。
4.3 应用查看时展示：
- Agent 与组件的状态。
- 连接信息（如 Endpoint、Service 名称）。

### 5. CRUD 与权限

5.1 Agent 实例：创建 / 更新 / 删除通过统一异步任务模型执行；查询同步执行。
5.2 Agent 应用：创建 / 更新 / 删除通过统一异步任务模型执行；查询同步执行。
5.3 组件实例：创建 / 更新 / 删除通过统一异步任务模型执行；查询同步执行。
5.4 权限模型：

- 超级管理员可操作全部工作空间与所有资源。
- 用户仅可操作其关联工作空间内资源。
5.5 工作空间内资源权限：
- 同一工作空间内用户可对该工作空间所有资源进行 CRUD。
5.6 工作空间隔离要求：
- 以工作空间为边界进行资源与权限隔离（namespace 映射 + RBAC + 跨边界关联限制）。
- 当前需求不承诺网络层默认隔离（如 NetworkPolicy Baseline），相关能力后续单独立项。
5.7 跨边界限制：
- 不允许跨 namespace 关联组件实例。
- 同一应用不可跨 K8S 集群部署，组件不可跨集群关联。
5.8 应用删除策略：
- 默认不级联删除关联组件实例。
- 支持用户显式选择是否级联删除关联组件实例。
- 当用户选择级联删除时，级联范围包含：关联组件实例、Agent 实例、PVC、Secret、ConfigMap（及与应用绑定的派生资源）。
5.9 异步任务模型（生产者-消费者）统一要求：
- 控制面作为生产者：接收 Agent 实例、组件实例、Agent 应用的 CUD 请求后，写入统一任务队列。
- 任务执行器作为消费者：异步拉取并执行任务，不在提交接口中阻塞等待资源最终状态。
- CUD 请求提交成功后，接口返回 `task_id`，由任务状态接口反馈执行进度与结果。
- 任务状态至少包含：`Pending`、`Running`、`Succeeded`、`Failed`、`Canceled`。
- 任务结果至少包含：资源标识、开始时间、结束时间、失败原因、重试次数。
- 任务执行应支持幂等控制、失败重试与并发控制（同一资源的任务需串行化）。
5.10 查询接口统一要求：
- Query 请求同步返回，不进入任务队列。
- 查询结果需包含最近一次异步任务状态与资源当前状态，用于前端统一展示。

## 需要澄清的问题

- 当前版本（截至 2026-02-19）无待澄清问题。

## 非功能需求

- 安全性：敏感信息（如密码）需保存在 Secret。
- 可观测性：提供部署日志、事件、状态查询。
- 可扩展性：支持新增平台组件模板与版本迭代。
- 可维护性：模板与配置分离，参数有校验规则。
- 验收与测试：验收标准与测试用例由需求方提供。
- 需求管理：需求持续演进，不设置冻结点；变更需同步更新需求与决策记录。

## 交付物与接口（建议）

- Agent 实例接口（CRUD）。
- 组件模板管理接口（读/列表/版本信息）。
- 组件实例接口（CRUD）。
- Agent 应用接口（CRUD）。
- 应用关系查询接口（应用 -> 组件关系）。
- 异步任务接口（任务状态查询 / 任务结果查询）。
